# PR16

## 这次提交在做什么

这批修改不是单点修补，而是在把 Naga 从“单主角对话应用”继续推进到“多干员 + 内嵌 OpenClaw runtime + 可打包桌面端”的形态。结合当前工作树，我把主线理解为以下几类：

1. 干员体系继续成型
新增了 `apiserver/agent_directory.py`，并在后端与前端之间补了干员目录、干员转发、干员会话相关能力。项目正在从单一娜迦主实例，扩展为：
- 内建 `naga-core` 干员
- 外挂 `openclaw` 干员
- 通讯录/目录化的多实例协作

2. Agentic Tool Loop 能力继续增强
`apiserver/agentic_tool_loop.py` 明显增强了本地工具执行和工具事件透传逻辑。它不再只是“解析工具 JSON 然后调用”，而是在承担：
- 工具调用解析
- 本地工具直执行
- 干员转发
- Live2D 联动
- 前端工具状态流同步

3. 打包链路继续收敛
`build.py`、`scripts/build-win.py`、`naga-backend.spec`、`frontend/electron-builder.yml`、`frontend/electron/modules/backend.ts` 这一组文件表明，项目正在把“开发态能跑”收敛为“桌面安装包稳定交付”：
- 版本号统一
- OpenClaw runtime 预装
- Electron 与后端启动流程统一
- 资源打包与平台差异处理继续补齐

4. Electron + Vue 前端继续从单窗口应用升级为桌面壳
从 `frontend/electron/*` 和 `frontend/src/*` 的改动看，前端已经不仅是页面集合，而是在承担：
- 启动阶段控制
- Live2D 资源切换
- 工具状态轮询
- 干员通讯录与多会话 UI
- 更新器、窗口模式、托盘/悬浮态联动

5. 角色系统和 prompt/source-of-truth 正在被理顺
`system/prompts/README.md`、`system/prompts/agentic_tool_prompt.txt`、`characters/README.md`、`system/character_bundle.py` 的方向很清楚：
- 角色模板
- 干员身份 bundle
- `.naga`/实例级成长
- OpenClaw 消费的提示词装配结果

这说明项目在从“提示词散落在多个地方”走向“角色、干员、运行时装配三层分离”。

## 我对项目的当前理解

### 1. 项目定位

NagaAgent 不是单纯聊天壳子。它本质上是一个桌面形态的 AI Agent 平台，核心目标是把以下几件事统一到一个产品里：

- LLM 对话
- 工具调用
- 本地/外部智能体协作
- Live2D 角色呈现
- 语音输入输出
- 社区和内容分发
- 桌面打包分发

也就是说，它同时是：
- AI 对话产品
- Agent orchestration 产品
- 桌面应用
- 角色驱动产品

### 2. 当前核心架构

我现在把项目理解成四层：

1. 表现层
- `frontend/src/*`
- `frontend/electron/*`

职责：
- Vue UI
- Electron 桌面容器
- 启动流程、窗口管理、更新、托盘、资源协议

2. API / 编排层
- `apiserver/*`

职责：
- 对话路由
- Agentic Tool Loop
- 对模型输出做工具解析、分发和回注
- 连接前端 UI 与后端能力

3. Agent / OpenClaw 层
- `agentserver/*`
- `agentserver/openclaw/*`
- `vendor/openclaw/*`

职责：
- OpenClaw runtime 管理
- Gateway 启停
- 干员实例管理
- 会话桥接
- 本地工具与外部 agent 执行

4. 配置 / 角色 / 记忆层
- `system/*`
- `characters/*`
- `summer_memory/*`

职责：
- 全局配置
- 角色模板与角色技能
- 提示词拼装
- 记忆系统
- 实例成长与人格边界

### 3. 现在最重要的产品方向

结合这次改动，我认为项目当前最重要的主轴不是“再加一个功能”，而是下面三件事：

1. 干员化
把“娜迦”从唯一主体变成系统中的一个内建干员，同时允许 OpenClaw 干员、角色模板干员、后续更多 agent 并存。

2. 打包化
把运行时依赖收敛进安装包，减少用户环境差异，尤其是 OpenClaw、Node、uv、前后端联动、资源路径问题。

3. 运行时一致性
开发环境、打包环境、角色资源、Live2D、工具调用、会话状态都要尽量走统一路径，否则产品会在桌面分发阶段不断出边角问题。

## 这批改动里我认为最关键的几个模块

### `apiserver/agentic_tool_loop.py`

这是项目的大脑中枢之一。它控制：
- 模型输出如何解释
- 工具何时执行
- 结果如何回注
- Live2D 与 UI 何时联动

谁掌握这个文件，谁就基本掌握了 Naga 当前 agent 调度行为。

### `agentserver/openclaw/embedded_runtime.py`

这是桌面分发能不能稳定的关键文件之一。它决定：
- 内嵌 runtime 怎样找
- gateway 怎样起
- packaged/source runtime 怎样切
- 运行时命令、端口、环境变量如何落地

OpenClaw runtime 的问题，大概率最终都要回到这里。

### `agentserver/openclaw/instance_manager.py`

这是“多干员”能不能真正成立的关键。只要项目继续走多 agent 方向，这个文件会越来越核心。

### `frontend/electron/main.ts`

这是桌面壳真正的入口。协议处理、资源加载、窗口、启动顺序、与后端关系都在这里汇总。桌面端很多看似“前端 bug”的问题，根子其实会在这里。

### `build.py`

这已经不是普通脚本，而是发布链路的编排器。它把版本同步、runtime 准备、前后端构建、平台差异处理串成了一个统一流程。

## 我现在对项目的结论

这个项目已经明显进入“产品工程化”阶段，不再是早期原型：

- 前端不是单独的 UI，而是桌面操作层
- 后端不是普通 API，而是 agent 编排层
- OpenClaw 不是可选插件，而是运行时核心能力之一
- 角色系统不是文案层，而是产品结构的一部分
- 打包不是最后一步，而是运行时设计的一部分

如果后续继续推进，我会优先关注：

1. 多干员的状态边界是否清晰
2. 打包态与开发态是否真正同构
3. OpenClaw runtime 生命周期是否继续稳定
4. 角色模板、角色 bundle、实例成长三层是否继续解耦
5. 前端是否持续用统一状态模型承接这些运行时能力
